# AGENTS.md
## Scope
- This repository is a FluxCD-managed Kubernetes GitOps repo.
- The primary entrypoint is `clusters/my-cluster`.
- Most work here is YAML-only; optimize for safe manifest edits and validation.
- Prefer small patches that preserve existing resource ordering and naming.

## Repo Rules
- No repo-local Cursor rules were found in `.cursor/rules/`.
- No `.cursorrules` file was found.
- No Copilot instructions file was found at `.github/copilot-instructions.md`.
- If any of those files are added later, treat them as higher-priority guidance and update this file.

## Layout
- `clusters/my-cluster/kustomization.yaml` is the top-level cluster overlay.
- Each app or subsystem usually lives in `clusters/my-cluster/<name>/`.
- `clusters/my-cluster/infrastructure/` wires ordered Flux `Kustomization` resources.
- `clusters/my-cluster/flux-system/` contains Flux bootstrap manifests.
- `clusters/my-cluster/cnpg-operator/cnpg-1.28.1.yaml` is vendored upstream content.

## Tooling
- Required CLIs: `kubectl` with built-in `kustomize`, and `flux`.
- On this machine, `kubectl` and `flux` are already installed.
- There is no repo-local `flake.nix`, `Makefile`, `Taskfile`, `package.json`, `go.mod`, or `pyproject.toml`.
- There is no dedicated lint runner or unit-test framework checked into this repository.
- If a command is missing elsewhere, use `nix shell` to provide it.

## Working Model
- Treat render and dry-run validation as the local build/test loop.
- Treat Flux reconciliation and status checks as integration verification.
- Validate the smallest affected scope first, then the full cluster overlay if the change is cross-cutting.
- Do not introduce Helm CLI workflows; this repo manages Helm through Flux `HelmRelease` resources.

## Build Commands
- Whole cluster render:
```bash
kubectl kustomize clusters/my-cluster >/dev/null
```
- Render a single app overlay:
```bash
kubectl kustomize clusters/my-cluster/harbor >/dev/null
```
- Render a single app for inspection:
```bash
kubectl kustomize clusters/my-cluster/sciflow
```
- Validate one raw manifest file:
```bash
kubectl apply --dry-run=client -f clusters/my-cluster/harbor/helmrelease.yaml
```

## Lint And Validation Commands
- There is no standalone linter configured; use Kubernetes client validation.
- Validate the full cluster render:
```bash
kubectl kustomize clusters/my-cluster | kubectl apply --dry-run=client -f -
```
- Validate one app overlay:
```bash
kubectl kustomize clusters/my-cluster/harbor | kubectl apply --dry-run=client -f -
```
- Validate one manifest file directly:
```bash
kubectl apply --dry-run=client -f clusters/my-cluster/cloudflared/helmrelease.yaml
```
- When editing Flux layering, also render the parent overlay that references it.

## Test Commands
- No conventional unit tests exist in this repo.
- The practical test loop is render -> dry-run -> reconcile -> inspect status.
- Full repo validation after a change:
```bash
kubectl kustomize clusters/my-cluster >/dev/null && \
kubectl kustomize clusters/my-cluster | kubectl apply --dry-run=client -f - >/dev/null
```

- Single-app validation after a change:

## Flux Commands
- Reconcile the Git source:
```bash
flux reconcile source git flux-system -n flux-system
```
- Reconcile the main cluster `Kustomization`:
```bash
flux reconcile kustomization flux-system -n flux-system --with-source
```
- Inspect cluster-level Flux state:
```bash
flux get all -A
```
- Watch recent events when debugging rollouts:
```bash
kubectl get events -A --sort-by=.lastTimestamp
```

## Secrets And Sensitive Data
- Never replace placeholders with real credentials in committed plaintext.
- Existing secrets use plain `Secret` manifests with placeholder values; preserve that pattern unless the repo adopts SOPS later.
- Keep `stringData` keys stable when workloads reference them through `valuesFrom` or environment imports.
- If you add a secret, include a placeholder and a helpful description annotation when useful.
- Do not rename secret keys casually; workloads and charts may depend on exact key names.

## Formatting
- Use two-space YAML indentation only; never use tabs.
- Keep key ordering stable unless a change clearly improves readability.
- Prefer one resource per file unless the file is generated or vendored.
- Keep lines reasonably compact, but favor readability over aggressive wrapping.
- Preserve blank-line patterns already used in the touched file.
- Quote strings only when YAML parsing or downstream consumers require it.

## Structure Conventions
- Start manifests with `apiVersion`, then `kind`, then `metadata`, then `spec` or `data`.
- Always set `metadata.name`.
- Set `metadata.namespace` for namespaced resources instead of relying on ambient context.
- In `kustomization.yaml`, keep `resources:` ordered intentionally and group dependencies before dependents.
- Put `Namespace`, `Secret`, `ConfigMap`, `HelmRepository` or `OCIRepository`, then `HelmRelease` or workloads in that order when practical.
- For Flux `Kustomization` resources, keep `sourceRef`, `path`, `dependsOn`, and prune behavior explicit when needed.

## Imports And References
- In this repo, "imports" are usually `resources:`, `valuesFrom:`, `envFrom:`, `sourceRef:`, and `chartRef:` relationships.
- Prefer relative Kustomize resource paths within the same directory.
- Keep `valuesFrom` entries aligned with the structure of the target chart values.
- Keep `sourceRef` namespaces explicit when the source lives in `flux-system`.
- When adding a new component directory, wire it into the nearest parent `kustomization.yaml` in dependency order.

## Types And API Usage
- Use the most specific Kubernetes kind already established here.
- Match the apiVersion already used by neighboring manifests unless performing an intentional upgrade.
- Do not mix alternate CRD versions within one feature area without a reason.
- When bumping chart or CRD versions, change only the minimum related fields and validate the affected overlay.

## Naming
- Directory names are lowercase kebab-case.
- Resource names are lowercase kebab-case.
- Namespace names are explicit and lowercase.
- Secret keys and environment variable names are uppercase with underscores when the application expects env-style names.
- Keep file names descriptive: `helmrelease.yaml`, `helmrepository.yaml`, `namespace.yaml`, `secret.yaml`, `localhost-ingress.yaml`, and similar.

## Comments
- Keep comments sparse and useful.
- Prefer comments for non-obvious operational constraints, ordering requirements, or placeholder warnings.
- Remove stale comments when changing behavior.
- Do not add narration comments for obvious YAML structure.

## Error Handling And Safety
- Fail closed: prefer placeholder secrets, explicit namespaces, and explicit source references over implicit behavior.
- When changing dependencies, verify parent overlays still render.
- When editing authentication, ingress, storage, or database settings, validate the single app and then the full cluster overlay.
- If a client-side dry-run cannot validate a CRD-backed field fully, still render locally and then use Flux and `kubectl describe` to confirm in-cluster behavior.
- Do not silently change persistence sizes, storage classes, hostnames, or image registries unless the task requires it.

## Change Strategy
- Make the smallest change that satisfies the request.
- Preserve unrelated user edits; the worktree may already be dirty.
- Prefer updating existing manifests over introducing duplicate resources.
- Reuse existing patterns from neighboring app directories before inventing new structure.
- If adding a new app, follow the established directory pattern: namespace, source, release or workload, ingress, then `kustomization.yaml`.

## Verification Checklist
- Render the smallest changed overlay.
- Dry-run the smallest changed overlay.
- If shared infrastructure changed, render and dry-run `clusters/my-cluster`.
- If Flux wiring changed, inspect `flux get all -A` after reconciliation.
- If workload behavior changed, inspect pods, events, and the owning `HelmRelease` or `Kustomization`.
